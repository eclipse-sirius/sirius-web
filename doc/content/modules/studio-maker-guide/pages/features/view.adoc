== Introduction

The View editor consumes the domain you modeled earlier and lets you describe one or more representations.
View models live inside *Studio projects*, just like domains, and they can be edited live without restarting the application.
Each view is an EMF model under the hood, so advanced users can still version it, compare it, or extend it programmatically, but studio makers only interact with the graphical editor.

== Live deployment model

* Studio makers author, test, and validate their views in the same running instance as end usersâ€”no code generation or packaging loop is required during design time.
* Once satisfied, export the studio project or package it into your {product} based application; the same view definition is loaded at runtime.
* Because views are stored as models, they can be processed by EMF tooling (compare, merge, document generation) if your team needs deeper governance.

== Best practices

* Keep representations focused: one view per persona or task prevents cluttered palettes.
* Document the intent of each mapping or tool using descriptions in the editor; those notes help future maintainers.
* Reuse color/text palettes across representations to ensure consistent branding.
* Combine the Query View or any scratch project to validate complex expressions before you wire them into styles or mappings.
